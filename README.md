## 1. ที่มาและความสำคัญ (Background & Significance)

ในยุคปัจจุบัน พฤติกรรมผู้บริโภคเปลี่ยนจากการซื้อสินค้าหน้าร้านมาสู่การซื้อขายผ่านช่องทางออนไลน์อย่างต่อเนื่อง โดยเฉพาะกลุ่มสินค้าแฟชั่นและเสื้อผ้าที่มีการแข่งขันสูง ผู้บริโภคคาดหวังประสบการณ์การช็อปปิ้งที่รวดเร็ว ใช้งานง่าย และมีความน่าเชื่อถือ อย่างไรก็ตาม แพลตฟอร์ม E-Commerce จำนวนมากในท้องตลาดยังคงมีข้อจำกัดหลายประการ เช่น

- หน้าเว็บโหลดช้า ส่งผลต่ออัตราการตีกลับ (Bounce Rate) และยอดขาย
- ระบบค้นหาสินค้าที่ไม่แม่นยำ ทำให้ลูกค้าหาสินค้าที่ต้องการไม่เจอ
- ขั้นตอนการชำระเงินซับซ้อนหลายขั้นตอนเกินไป
- ขาดระบบติดตามและแก้ไขปัญหาหลังการขายที่เป็นระบบ

## 2. Pain Points (ปัญหาที่พบในปัจจุบัน) — รายละเอียดเชิงลึก

### 2.1 ปัญหาฝั่งลูกค้า (Customer Pain Points)

| ปัญหา | ผลกระทบ | สาเหตุ |
|---|---|---|
| ค้นหาสินค้าไม่เจอ / ผลลัพธ์ไม่ตรงกับที่ต้องการ | ลูกค้าเสียเวลา เกิดความหงุดหงิด อาจปิดเว็บไปซื้อที่อื่น | ระบบค้นหาไม่มี Filter ตามไซส์/สี/ราคา หรือไม่มี Search Index ที่ดี |
| ขั้นตอนชำระเงินซับซ้อน | อัตราการทิ้งตะกร้าสินค้า (Cart Abandonment) สูง | ฟอร์มกรอกข้อมูลยาว ไม่มีการบันทึกที่อยู่/บัตรไว้ล่วงหน้า |
| ดูประวัติการซื้อย้อนหลังไม่ได้ | ลูกค้าไม่สามารถติดตามสถานะพัสดุ หรืออ้างอิงเพื่อขอคืนสินค้า | ไม่มีระบบ Order History ที่เชื่อมโยงกับบัญชีผู้ใช้ |
| เมื่อสินค้ามีปัญหา (เช่น ส่งผิดไซส์) ไม่มีช่องทางแจ้งที่เป็นระบบ | ต้องทักแชทเพจที่ตอบช้า เสียเวลารอ ความไม่พึงพอใจสะสม | ไม่มีระบบ Ticket ที่ติดตามสถานะได้ |
| ไม่มีการแจ้งเตือนสถานะคำสั่งซื้อ | ลูกค้าไม่ทราบว่าออเดอร์อยู่ขั้นตอนไหน | ขาดระบบ Notification |

### 2.2 ปัญหาฝั่งแอดมิน (Admin Pain Points)

| ปัญหา | ผลกระทบ | สาเหตุ |
|---|---|---|
| ไม่มีเครื่องมือจัดการร้านค้าแบบจุดเดียว | ต้องสลับไปมาหลายระบบ เสียเวลาในการทำงาน | ไม่มี Centralized Dashboard |
| เพิ่ม/ลด/แก้ไขสินค้ายาก | อัปเดตสต็อกช้า สินค้าหมดแต่ยังแสดงขายอยู่ (Overselling) | ไม่มีระบบ CRUD สินค้าที่ใช้งานง่าย |
| จัดการหมวดหมู่สินค้าไม่เป็นระบบ | สินค้าจัดหมวดหมู่สับสน ลูกค้าหาไม่เจอ | ไม่มี Category Management ที่ยืดหยุ่น |
| จัดการข้อมูลลูกค้าลำบาก | วิเคราะห์พฤติกรรมลูกค้าเพื่อทำการตลาดได้ยาก | ไม่มีระบบ Customer Management ที่รวมศูนย์ |
| ไม่มีข้อมูลสรุปภาพรวมร้านค้า | ตัดสินใจทางธุรกิจช้า ขาดข้อมูลประกอบการตัดสินใจ | ไม่มี Dashboard สรุปยอดขาย/สต็อก |

### 2.3 ปัญหาฝั่งบริการลูกค้า (Support Pain Points)

| ปัญหา | ผลกระทบ | สาเหตุ |
|---|---|---|
| ไม่มีระบบ Ticket | สับสนว่าเคสไหนแก้แล้ว เคสไหนค้างอยู่ | ใช้การสื่อสารผ่านแชทที่ไม่มีการบันทึกสถานะ |
| ไม่มีการจัดลำดับความสำคัญของเคส | เคสเร่งด่วนอาจถูกละเลย | ไม่มีระบบ Priority/SLA |
| ไม่มีประวัติการสนทนาที่เชื่อมกับออเดอร์ | ต้องถามข้อมูลซ้ำจากลูกค้า เสียเวลา | Ticket ไม่ผูกกับ Order ID |
| ทีมซัพพอร์ตหลายคนอาจตอบเคสเดียวกันซ้ำซ้อน | สร้างความสับสนให้ลูกค้า | ไม่มีระบบมอบหมายงาน (Assignment) |

## 3. วัตถุประสงค์ (Objectives)

1. เพื่อพัฒนาเว็บไซต์ E-Commerce แบรนด์เสื้อผ้า SALA ที่ใช้งานง่าย รองรับการทำงานในทุกอุปกรณ์ (Responsive Design) ทั้งมือถือ แท็บเล็ต และคอมพิวเตอร์
2. เพื่อสร้างระบบการซื้อขาย ตะกร้าสินค้า และประวัติการสั่งซื้อที่มีประสิทธิภาพ ปลอดภัย และโปร่งใส สามารถตรวจสอบย้อนหลังได้
3. เพื่อพัฒนาระบบ Customer Support ในรูปแบบ Ticket ที่เชื่อมต่อระหว่างลูกค้าและเจ้าหน้าที่ได้อย่างไร้รอยต่อ พร้อมระบบติดตามสถานะ
4. เพื่อสร้าง Dashboard หลังบ้านที่ช่วยให้ผู้ดูแลระบบบริหารจัดการสินค้า หมวดหมู่ และข้อมูลลูกค้าได้ในที่เดียว
5. เพื่อออกแบบสถาปัตยกรรมระบบที่รองรับการขยายตัว (Scalable Architecture) ในอนาคต ทั้งด้านจำนวนผู้ใช้งานและปริมาณธุรกรรม
6. เพื่อสร้างความปลอดภัยของข้อมูลลูกค้าและธุรกรรมทางการเงินตามมาตรฐานสากล

## 4. Scope (ขอบเขตของระบบ)

### 4.1 สิ่งที่อยู่ในขอบเขต (In Scope)

- ระบบสมัครสมาชิก / เข้าสู่ระบบ (Authentication)
- ระบบแสดงสินค้า ค้นหา และกรองสินค้า
- ระบบตะกร้าสินค้าและ Checkout
- ระบบชำระเงิน (เชื่อมต่อ Payment Gateway)
- ระบบประวัติการสั่งซื้อ
- ระบบ Ticket แจ้งปัญหา/ขอความช่วยเหลือ
- Admin Dashboard สำหรับจัดการสินค้า หมวดหมู่ และลูกค้า
- Support Workspace สำหรับจัดการ Ticket
- ระบบแจ้งเตือน (Notification) พื้นฐาน เช่น อีเมลยืนยันคำสั่งซื้อ


### 4.2 สิ่งที่อยู่นอกขอบเขต (Out of Scope)
- ระบบแนะนำสินค้าด้วย AI/Machine Learning (Recommendation Engine)
- แอปพลิเคชันมือถือแบบ Native (iOS/Android)
- ระบบ Live Chat แบบ Real-time กับ Support
- ระบบ Loyalty Point / สมาชิกสะสมแต้ม
- ระบบ Marketplace หลายผู้ขาย (Multi-vendor)

> หมายเหตุ: รายการนอกขอบเขตสามารถพิจารณาเพิ่มเติมได้ในเฟสถัดไปของโปรเจกต์

## 5. ผู้ใช้งาน (Users) และบทบาท

ระบบแบ่งผู้ใช้งานออกเป็น 3 กลุ่มหลัก:

### 5.1 Customer (ลูกค้า)
ผู้ใช้งานทั่วไปที่เข้ามาเลือกซื้อสินค้า มีสิทธิ์จำกัดเฉพาะข้อมูลของตนเอง


### 5.2 Admin (ผู้ดูแลระบบ)
ผู้บริหารจัดการร้านค้า มีสิทธิ์เต็มในการจัดการสินค้า หมวดหมู่ และข้อมูลลูกค้า รวมถึงดูภาพรวมสถิติร้านค้า


### 5.3 Support (เจ้าหน้าที่ดูแลลูกค้า)
ผู้รับผิดชอบดูแล Ticket ที่ลูกค้าส่งเข้ามา ตอบกลับและแก้ไขปัญหาให้ลูกค้า

## 7. System Requirements

## 7.1 Functional Requirements (FR)

| รหัส | Module | รายละเอียด |
|---|---|---|
| FR-01 | Authentication Service | ระบบ Sign-up, Log-in, Log-out, Reset Password, Password Hashing (bcrypt) |

| FR-03 | Search Engine | ค้นหาสินค้าด้วยคำสำคัญ พร้อม Filter (หมวดหมู่/ไซส์/สี/ราคา) และ Sort (ราคา/ความนิยม/ใหม่ล่าสุด) |
| FR-04 | Shopping Cart | เพิ่ม/ลบ/แก้ไขจำนวนสินค้าในตะกร้า คำนวณราคารวมอัตโนมัติ |
| FR-05 | Checkout Engine | ยืนยันที่อยู่จัดส่ง เลือกวิธีชำระเงิน คำนวณค่าจัดส่ง ตัดสต็อกเมื่อชำระเงินสำเร็จ |
| FR-06 | Payment Integration | เชื่อมต่อ Payment Gateway (เช่น บัตรเครดิต, พร้อมเพย์, TrueMoney) |
| FR-07 | Order Tracking | บันทึกและแสดงประวัติคำสั่งซื้อ พร้อมสถานะการจัดส่ง |
| FR-08 | Ticket Generation | ลูกค้าสร้าง Ticket ผูกกับ Order ID พร้อมแนบไฟล์ |
| FR-09 | Ticket Management | Support ดู/ตอบ/เปลี่ยนสถานะ/มอบหมาย Ticket |
| FR-10 | Admin Product CRUD | Admin เพิ่ม/แก้ไข/ลบสินค้าและหมวดหมู่ |
| FR-11 | Admin Customer Management | Admin ดู/ค้นหา/จัดการสถานะบัญชีลูกค้า |
| FR-12 | Admin Dashboard & Analytics | แสดงกราฟยอดขาย สินค้าขายดี สินค้าคงเหลือ จำนวน Ticket คงค้าง |
| FR-13 | Notification Service | ส่งอีเมล/แจ้งเตือนยืนยันคำสั่งซื้อ และอัปเดตสถานะ Ticket |

## 7.2 Non-Functional Requirements (NFR)

| รหัส | หมวดหมู่ | รายละเอียด |
|---|---|---|
| NFR-01 | Performance | หน้าเว็บหน้าร้านต้องแสดงผลและโหลดข้อมูลเสร็จสิ้นภายใน 2 วินาที ภายใต้ Load ปกติ |
| NFR-02 | Security | เข้ารหัสรหัสผ่านด้วย bcrypt, ใช้ HTTPS ทุก Endpoint, ป้องกัน SQL Injection/XSS/CSRF, ข้อมูลบัตรเครดิตไม่จัดเก็บโดยตรง (ใช้ Tokenization ผ่าน Payment Gateway) |
| NFR-03 | Scalability | ระบบ Ticket และระบบสั่งซื้อออกแบบแยก Service (Microservice-ready) เพื่อรองรับปริมาณเคส/ออเดอร์ที่เพิ่มขึ้น |
| NFR-04 | Usability | ดีไซน์มินิมอล ใช้งานง่าย รองรับ Responsive ทั้งมือถือและคอมพิวเตอร์ |
| NFR-05 | Reliability | ระบบต้องมี Uptime ไม่ต่ำกว่า 99.5% ต่อเดือน |
| NFR-06 | Maintainability | โค้ดต้องเป็นไปตามมาตรฐาน (Coding Standard), มีเอกสารประกอบ API (Swagger/OpenAPI) |
| NFR-07 | Data Backup & Recovery | สำรองข้อมูลอัตโนมัติทุกวัน และสามารถกู้คืนข้อมูลได้ภายในเวลาที่กำหนด (RTO/RPO) |
| NFR-08 | Compatibility | รองรับ Browser หลัก (Chrome, Safari, Edge, Firefox) เวอร์ชันล่าสุด |
| NFR-09 | Accessibility | ปฏิบัติตามแนวทาง WCAG เบื้องต้น เช่น Contrast สี, Alt text รูปภาพ |

## 8. เทคโนโลยีที่ใช้ (Technology Stack)

โปรเจกต์ SALA เลือกใช้เทคโนโลยีที่มีความเสถียร เป็นที่นิยมในวงกว้าง และมีเอกสาร/Community รองรับสูง เพื่อให้ทีมพัฒนาสามารถทำงานร่วมกันได้อย่างมีประสิทธิภาพ และลดความเสี่ยงในการดูแลรักษาระบบในระยะยาว

| เทคโนโลยี | บทบาทในระบบ |
|---|---|
| **React.js | เฟรมเวิร์กหลักในการสร้าง User Interface

|---|---|
| **MySQL** | ฐานข้อมูลเชิงสัมพันธ์ (Relational Database) สำหรับจัดเก็บข้อมูลที่มีความสัมพันธ์ซับซ้อน เช่น สินค้าและตัวเลือกสต็อกแยกตามไซส์/สี, คำสั่งซื้อและรายการสินค้าในออเดอร์, รวมถึงระบบ Ticket ที่ต้องเชื่อมโยงกับผู้ใช้และคำสั่งซื้อ ความสามารถด้าน Transaction (ACID) ช่วยรับประกันความถูกต้องของข้อมูล เช่น การตัดสต็อกเมื่อชำระเงินสำเร็จ |

## 9. Database Design (โครงสร้างฐานข้อมูลเบื้องต้น)

### 9.1 ตารางหลัก (Core Tables)

- **users** — เก็บข้อมูลผู้ใช้งานทั้ง 3 บทบาท (customer, admin, support)
- **addresses** — ที่อยู่จัดส่งของลูกค้า (1 user มีได้หลายที่อยู่)
- **categories** — หมวดหมู่สินค้า
- **products** — ข้อมูลสินค้า
- **product_variants** — ตัวเลือกสินค้า (ไซส์/สี/สต็อกแยกตามตัวเลือก)
- **carts** — ตะกร้าสินค้าของแต่ละผู้ใช้
- **cart_items** — รายการสินค้าภายในตะกร้า
- **orders** — คำสั่งซื้อ
- **order_items** — รายการสินค้าภายในคำสั่งซื้อ
- **payments** — ข้อมูลการชำระเงิน
- **tickets** — คำร้อง/แจ้งปัญหาจากลูกค้า
- **ticket_messages** — ข้อความสนทนาภายใน Ticket
